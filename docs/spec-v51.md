# spec-v51.md — Minimal homepage: search + 10 curated quick picks

> Status: proposed (2026-05-23). v51 is a UI / SEO spec. It
> redesigns the public homepage at `https://sophiewell.com/` to
> the smallest useful surface for a nurse at the bedside: a
> header, a one-line title, a one-paragraph lede, a single
> search input, ten curated quick picks, and a footer. Every
> other homepage element — the trust strip, the audience-hub
> link strip, the audience filter chips, the full browse-all
> tile grid, the "Why Sophie" prose, and the visible FAQ — is
> removed from the home view.
>
> v51 adds **zero clinical tiles.** Catalog count is unchanged.
> Every prior spec (v4–v50) remains in force.

## 1. Why v51 exists

Sophie's wedge is "nurse opens a browser at the bedside, types
the score they need, gets the answer." The closer the homepage
is to *one search box*, the faster that wedge fires. The prior
homepage carried five competing surfaces (trust strip, hub
strip, audience chips, browse disclosure with 254 tiles, "Why
Sophie" prose, visible FAQ) — each useful in isolation, each
adding cognitive load at the moment when the nurse cares about
exactly one thing: typing the tool name.

The maintainer's framing on 2026-05-23 was explicit: "I want
this UI to be INCREDIBLY clean… ideally, I want it to ONLY be
a search bar after the title explaining what sophiewell does."
v51 codifies that decision and locks it against drift.

## 2. Non-goals

- **No tile removal.** The 254-tile catalog is preserved. Tiles
  are still individually addressable at `/tools/<id>/` and grouped
  on the audience hubs at `/for/<slug>/`.
- **No SEO loss.** Every visible-on-old-homepage SEO surface
  (FAQPage answers, WebApplication feature list, audience-hub
  links) survives in one of three forms:
  1. JSON-LD `FAQPage` + `WebApplication` blocks in `<head>`
     (unchanged from [spec-seo](spec-seo.md); regenerated by
     `scripts/build-ld.mjs`).
  2. Real routes (`/tools/<id>/`, `/for/<slug>/`,
     `/topics/<slug>/`, `/commitments/`) listed in
     `sitemap.xml` via `scripts/build-sitemap.mjs`.
  3. The minimal `home-lede` paragraph, which retains the
     "254 deterministic calculators…" sentence for
     `check-catalog-truth.mjs` and SERP-snippet purposes.
- **No analytics.** "Most-used" quick picks are curated, not
  measured. Sophie ships no telemetry — [spec-v50](spec-v50.md)
  §3.5 prohibits it. The quick-picks list is a hand-edited
  array of ten tiles, reviewed when the catalog shifts.
- **No autocomplete dropdown / live tile rendering.** The
  search input filters nothing visible; on Enter it resolves
  the query against the synonym table + fuzzy ranker and
  routes to the matched tile. The hero-synonym-hint
  breadcrumb (existing) confirms the match before the user
  presses Enter.

## 3. Homepage contract (v51)

Inside `index.html`, between `<header class="topbar">` and
`<footer class="site-footer">`, the homepage MUST consist of
exactly the following blocks, in order:

1. `<h1 class="home-h1">` — one-line product summary
   ("Free healthcare tools, all in your browser").
2. `<p class="home-lede">` — one paragraph leading with the
   catalog count and the four trust phrases ("No signup, no
   tracking, no AI").
3. `<section id="home-view">` containing:
   a. `<h2 id="tools-heading" class="visually-hidden">` —
      the screen-reader landmark.
   b. `<div class="task-hero">` — `<label for="hero-search">`
      reading "Search NNN tools" (NNN matches
      `UTILITIES.length`) and the `<input id="hero-search">`.
   c. `<p id="hero-synonym-hint" hidden>` — the synonym-match
      breadcrumb (populated by `app.js`).
   d. `<section class="quick-picks">` — heading
      "Quick picks" and a `<div class="quick-picks-grid">`
      with exactly ten `<button class="quick-pick">` tiles.

No other visible block is permitted on the homepage. In
particular, the following MUST NOT appear:

- `.trust-strip`
- `.hub-strip`
- `.audience-chips` / any `.toggle-group`
- `#browse-disclosure` / `#tile-grid` / `#empty-state` /
  `#browse-tile-count`
- `.why-sophie`
- `.visible-faq`
- `#hero-examples`

A `grep-check` rule MAY enforce this in CI; until then the
prohibition is documented here and reviewed at PR time.

## 4. Quick picks

The ten quick-pick tiles are **curated**, not ranked. The
selection follows the nurse-first pivot
([spec-v29](spec-v29.md) §5.3): every quick pick is a tool a
floor / ICU / ED nurse can use at the bedside without an MD
sign-off step. Initial selection (2026-05-23):

| `data-tool` id   | Visible title           | Bedside use            |
|------------------|-------------------------|------------------------|
| `nihss`          | NIH Stroke Scale        | Acute stroke severity  |
| `chads`          | CHA2DS2-VASc            | AFib stroke risk       |
| `wells-pe`       | Wells Score for PE      | PE pretest probability |
| `heart`          | HEART Score             | Chest-pain MACE risk   |
| `qsofa-sofa`     | qSOFA / SOFA            | Sepsis screen + score  |
| `ciwa`           | CIWA-Ar                 | Alcohol withdrawal     |
| `gcs`            | Glasgow Coma Scale      | Neuro check            |
| `shock-index`    | MAP / Shock Index       | Hemodynamic snapshot   |
| `anion-gap-dd`   | Anion Gap & Delta-Delta | Acid-base workup       |
| `cssrs`          | C-SSRS Screener         | Suicide-risk screen    |

Changing the quick-picks list requires (a) editing
`index.html`, (b) noting the change in this spec's
changelog, and (c) confirming each `data-tool` id resolves
to an active entry in `UTILITIES` (build-time grep).

## 5. /commitments/ as the canonical commitments page

[spec-v50](spec-v50.md) §4 already ships
`/commitments/index.html` via
`scripts/build-commitments-page.mjs`. Prior to v51 the
homepage footer included a prominent link to it
(`.footer-commitments-link`). v51 removes that link as part
of the footer minimization (footer is now just the
"Made with ❤ by Clay Good" credit and the GitHub badge).

The /commitments/ page remains:

- A real, indexable route at `https://sophiewell.com/commitments/`.
- Listed in `sitemap.xml` (added by v51 to
  `scripts/build-sitemap.mjs`, priority 0.6, monthly
  changefreq).
- Referenced in its own `<head>` via canonical URL and
  breadcrumb JSON-LD.
- Reachable from inside the /commitments/ page itself via
  the existing "← Back to Sophie Well" footer link.

The maintainer's framing on 2026-05-23 was: "remove the
'Sophie's Commitments' [footer line] and just be the made
with ❤ by Clay Good and GitHub links/buttons." Discoverability
of /commitments/ now relies on:

- Direct URL navigation (the canonical URL is stable).
- The Google SERP for queries like "sophie well commitments"
  or "sophie well no tracking" — the page's `<title>` and
  `<description>` are SEO-shaped for these.
- The README and CONTRIBUTING.md, which link the page in
  prose.

If discoverability becomes a real problem (measured by zero
incoming traffic to /commitments/ at 6-month review), a
single footer link MAY be restored. The default is removed.

## 6. SEO posture

The minimum-viable homepage retains every SEO surface that
matters to crawlers:

- `<title>`, `<meta name="description">`, OG, Twitter — all
  unchanged, carry the "254 Tools" catalog count.
- JSON-LD `WebApplication` block with full 254-name
  `featureList` — unchanged, regenerated by
  `scripts/build-ld.mjs` from `UTILITIES`.
- JSON-LD `FAQPage` block with 12 Q/A entries — unchanged,
  regenerated by `scripts/build-ld.mjs`. The visible
  `<details>` mirror of this block is removed; Google reads
  the `FAQPage` schema from `<head>` regardless of whether
  the answers are mirrored in visible HTML.
- `sitemap.xml` enumerates `/`, `/commitments/`, `/topics/`,
  each `/for/<slug>/`, each `/topics/<slug>/`, and every
  `/tools/<id>/`. v51 adds `/commitments/`.
- Per-tool pages at `/tools/<id>/` — unchanged; remain the
  canonical landing surfaces for long-tail queries
  (e.g. "wells score for PE calculator").
- Audience hubs at `/for/<slug>/` — unchanged; remain the
  canonical landing surfaces for audience queries
  (e.g. "nurse calculators").

The trade-off accepted by v51: the homepage itself loses
some "long-tail relevance" because the 254 tile names no
longer appear as visible anchor text on `/`. The per-tool
pages absorb this — they already rank better for
tool-specific queries than the homepage ever did.

**Hidden content for SEO is explicitly disallowed.** v51 does
not move the removed content into `display:none` divs or
similar cloaked surfaces. The content is either (a) in
JSON-LD (which is a recognized SEO mechanism, not cloaking),
(b) on a real route (which crawlers reach via sitemap), or
(c) deleted entirely.

## 7. Checks and tests updated

The following automated surfaces were updated alongside the
markup:

- `scripts/check-catalog-truth.mjs` — replaced the
  `#browse-tile-count` surface with a new "hero search label"
  surface that reads `<label for="hero-search">Search NNN
  tools</label>`. The "home lede" surface is unchanged.
- `scripts/build-sitemap.mjs` — added `/commitments/` URL.
- `test/integration/smoke.spec.js` — the home assertion now
  checks for `#hero-search` + ten `.quick-pick` cards
  instead of 150+ `.tool-card` cards; the BMI route smoke
  navigates via `/#bmi` deep link instead of clicking through
  the browse disclosure.
- `test/integration/no-network.spec.js` and
  `test/integration/all-tools.spec.js` — removed the
  `document.getElementById('browse-disclosure').open = true`
  call (the element no longer exists).
- `test/unit/hash.test.js` — `parseHash` / `buildHash` for
  the `b=open|closed` key are unchanged (the key is parsed
  but no longer read by `app.js` at runtime; left intact for
  bookmark stability).

## 8. Migration & rollback

Rolling forward: a single PR replaces the homepage markup,
deletes the now-dead CSS for `.trust-strip`, `.hub-strip`,
`.audience-chips`, `.browse-disclosure`, `.why-sophie`,
`.visible-faq`, `.footer-commitments-link`, and updates the
four checks listed in §7. The dead CSS may be removed in a
follow-up wave (v51-2) since unused rules are inert.

Rolling back: revert the v51 PR. No data migration is
required; no URLs change shape; no `localStorage` keys are
added or removed (none of the removed elements wrote to
storage).
